# DDUK ERP 포트폴리오 프로젝트 전용 AI 개발 하네스

이 문서는 DDUK ERP 팀 프로젝트에서 AI가 일관된 기준으로 작업하도록 돕는 운영 가이드다.

프로젝트의 목적은 아래와 같다.

- 실무형 ERP 아키텍처 경험
- 협업 구조 경험
- 유지보수 가능한 코드 작성
- ERP 도메인 흐름 학습
- 안정적인 CRUD/API 설계
- 포트폴리오 완성도 향상

이 프로젝트는 실제 대기업 ERP 수준의 과도한 복잡성보다  
"실무스럽고 유지보수 가능한 구조"를 우선한다.

---

## 1. 프로젝트 구조 규칙

프로젝트 기본 구조는 아래를 따른다.

- `backend/` → Spring Boot 기반 ERP Core
- `frontend/` → HTML/CSS/Vanilla JS
- `ai-server/` → Flask 기반 AI API
- `rpa/` → Playwright 자동화
- `docs/` → 설계 및 문서

원칙:

- 새 최상위 디렉터리는 임의로 만들지 않는다.
- 가능하면 기존 구조를 재사용한다.
- 새 파일이나 하위 구조는 현재 도메인/기술 스택 흐름 안에서 추가한다.

---

## 2. 도메인 규칙

프로젝트 도메인은 아래 기준을 따른다.

- `hr`
  - 인사
  - 급여
  - 회계
- `inventory`
  - 재고
  - 발주
  - 입출고
  - 거래처
- `admin`
  - 로그인
  - 권한
  - 공통 기능
  - 시스템 설정

AI는 작업 요청 시:

- 먼저 어떤 도메인인지 판단한다.
- 해당 도메인 하위 구조를 우선 사용한다.
- 공통 기능은 `common`, `shared` 성격으로 분리할지 검토한다.

---

## 3. 핵심 개발 원칙

우선 원칙:

- 읽기 쉬운 코드
- 유지보수 가능한 구조
- 기존 구조 존중
- 과도한 추상화 지양
- ERP 흐름 유지
- 보안 기본 준수
- 협업 친화적 코드 작성

포트폴리오 프로젝트이므로 아래도 같이 지킨다.

- 지나치게 복잡한 엔터프라이즈 패턴 남용 금지
- 불필요한 마이크로서비스 구조 지양
- 과한 디자인 패턴 적용 지양
- 복잡성보다 명확한 구조 우선

---

## 4. 금지 및 주의 사항

아래는 원칙적으로 금지한다.

- 존재하지 않는 테이블, 컬럼, API를 실제 구현 사실처럼 단정하기
- 비밀번호 평문 저장
- Secret, API Key 하드코딩
- Controller에 대규모 비즈니스 로직 작성
- `SELECT *` 사용
- 권한 검증 생략
- 스택 트레이스나 내부 민감 오류를 API 응답으로 노출

아래는 현재 설계 수준에 맞게 판단한다.

- 테이블, 컬럼, API 초안 제안은 가능하다.
- 다만 제안일 경우 반드시 초안, 가정, 추정 중 하나로 명시한다.
- 현재 문서나 코드에 없는 구조를 새로 제안할 때는 근거와 영향 범위를 함께 설명한다.
- 불확실성이 큰 경우에는 질문하거나, 가능한 선택지를 짧게 제시한다.

---

## 5. Backend 규칙 (Spring Boot)

구조 원칙:

- Controller: 요청/응답 처리
- Service: 비즈니스 로직 처리
- Repository: DB 접근 처리
- DTO / Entity 역할 분리

네이밍:

- Class → PascalCase
- method / field → camelCase

개발 원칙:

- 가능하면 Service Layer 중심 구조를 유지한다.
- 트랜잭션 경계를 고려한다.
- 중복 코드를 줄인다.
- 기존 구조가 있으면 그 흐름을 우선 따른다.

단:

- 과도한 계층 분리는 지양한다.
- 작은 기능에 불필요한 추상화는 피한다.

---

## 6. Frontend 규칙

- HTML, CSS, Vanilla JS 기반을 유지한다.
- **파일 분리 원칙**: HTML, JS, CSS는 상호 분리하여 관리한다.
  - HTML 내부에 `<style>` 태그나 대규모 `<script>` 인라인 로직 작성을 지양한다.
  - 공통 로직은 `services/common/` 또는 `styles/common/`에 위치시키고, 페이지 전용 로직은 별도 파일로 분리한다.
- 기존 라우팅 구조를 우선 유지한다.
- UI 화려함보다 기능 안정성을 우선한다.
- 하드코딩 URL은 최소화한다.
- 페이지 구조와 코드 스타일은 일관되게 유지한다.

포트폴리오 목적상:

- 코드 가독성을 우선한다.
- 지나친 최적화보다 명확한 구조를 우선한다.

## 8. Database 규칙

MySQL 기준:

- `snake_case` 사용
- PK 기본명은 `id` 사용
- FK / index naming 일관성 유지

ERP 기본 규칙:

- 재고 음수 방지를 고려한다.
- 상태값 흐름을 유지한다.
- 주요 변경 로그 필요성을 검토한다.

팀 합의 규칙:

- DB 설계 전 아래 항목은 먼저 팀에서 합의한다.
  - 인증 기준 테이블을 무엇으로 둘지
  - 역할 값을 무엇으로 고정할지
  - 직원과 로그인 계정을 분리할지 연결할지
  - 거래처, 품목, 재고, 발주의 핵심 관계를 어떻게 둘지
  - 상태값을 어디까지 세분화할지
- 이미 코드에서 사용 중인 테이블명, 컬럼명, 상태값은 합의 없이 임의로 바꾸지 않는다.
- 컬럼 추가는 가능하지만, 기존 계약을 깨지 않는 방향으로 추가한다.
  - nullable 컬럼 또는 기본값 있는 컬럼을 우선한다.
  - `NOT NULL` 신규 컬럼은 기존 데이터, 시드 데이터, API 영향까지 같이 확인한다.
  - 계좌번호, 사업자번호, 전화번호처럼 형식이 고정되지 않은 값은 숫자형보다 문자열로 관리한다.
- 민감하거나 운영상 중요한 정보는 특히 더 조심한다.
  - 비밀번호, 토큰, 시크릿은 DB 초안이나 시드에 평문으로 두지 않는다.
  - 계좌정보, 연락처, 이메일은 화면/API 노출 범위까지 같이 본다.
- MVP 범위를 넘는 과설계는 미룬다.
  - 실제로 안 쓰는 권한 세분화 테이블
  - 아직 필요하지 않은 다단계 승인 구조
  - 아직 도입되지 않은 멀티테넌트 컬럼
- DB 변경 제안에는 가능하면 아래 내용을 같이 남긴다.
  - 왜 필요한지
  - 어느 도메인에 속하는지
  - API / 백엔드 / 프론트 영향 범위
  - migration 또는 시드 수정 필요 여부

Schema 변경 시:

- 영향 범위를 설명한다.
- migration 필요 여부를 함께 본다.

## 9. API 규칙

기본 prefix:

`/api/v1/`

RESTful 규칙을 기본으로 따른다.

응답 형식:

성공:

```json
{
  "status": "success",
  "data": {},
  "message": ""
}
```

실패:

```json
{
  "status": "error",
  "message": "",
  "code": ""
}
```

원칙:

- HTTP 상태코드는 의미에 맞게 사용한다.
- 민감한 내부 오류는 외부에 노출하지 않는다.
- API 초안이나 문서와 다른 구조를 제안할 때는 차이를 명시한다.

## 11. AI 응답 규칙

가능하면 사용자에게 보내는 최종 응답 첫 줄은 정확히 아래 문구로 시작한다.

`뚝 erp 하네스 적용 중!`

구현, 설계, 리뷰처럼 구조화가 필요한 요청에는 아래 형식을 우선 사용한다.

1. 요구사항 분석
2. 영향 범위
3. 구현 전략
4. 코드
5. 테스트 방법

단:

- 짧은 질답, 간단 확인, 상태 공유에는 더 간결하게 답변할 수 있다.
- 불확실한 내용은 `추정`이라고 명시한다.

## 13. AI 행동 제한

AI는 아래를 우선한다.

- 기존 코드 스타일 우선
- 기존 구조 우선
- 최소 수정 원칙 준수
- unrelated 변경 금지
- 임의 리팩토링 지양

코드 생성 시:

- 실행 가능한 코드 작성
- pseudo code 금지
- TODO 남발 금지

## 14. 멀티테넌트 규칙

현재 프로젝트에 실제 멀티테넌트 구조가 도입된 경우에만 아래를 강하게 적용한다.

- `tenant_id` 고려
- tenant 데이터 혼합 방지
- 권한 범위 고려

아직 멀티테넌트가 도입되지 않았다면:

- 향후 확장 가능성만 염두에 둔다.
- 존재하지 않는 `tenant_id`를 임의로 코드에 끼워 넣지 않는다.

---

## 15. ERP 상태 관리 규칙

ERP 문서는 상태 흐름을 가진다.

예:

`DRAFT`  
→ `APPROVED`  
→ `COMPLETED`

원칙:

- 허용되지 않은 상태 변경은 지양한다.
- 상태 변경 시 권한, 로그, 데이터 영향을 함께 본다.
- 아직 상태 전이 규칙이 확정되지 않았다면 초안으로 제안하고 확정 여부를 명시한다.

## 17. 운영 메모

이 하네스는 "AI를 묶어두는 절대 규칙집"이라기보다  
"팀 프로젝트를 실무스럽게 진행하기 위한 운영 기준"에 가깝다.

따라서:

- 현재 문서와 코드가 우선 기준이다.
- 문서가 비어 있는 부분은 초안 제안이 가능하다.
- 다만 초안은 구현 사실처럼 단정하지 않는다.
- 프로젝트 단계가 바뀌면 이 하네스도 같이 업데이트한다.
