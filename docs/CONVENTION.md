# DDUK ERP 협업 가이드라인 (Convention)

이 문서는 DDUK ERP 프로젝트의 팀원과 AI가 같은 기준으로 작업하기 위한 협업 규칙이다.

---

## 1. 프로젝트 구조
모든 개발은 아래 구조 안에서 진행한다. 새 최상위 폴더를 만들 때는 팀 합의가 필요하다.

- `backend/`: Spring Boot 기반 ERP 코어
- `frontend/`: HTML, CSS, Vanilla JS
- `ai-server/`: Flask 기반 AI API
- `rpa/`: Playwright 자동화 스크립트
- `docs/`: 설계, 정책, 컨벤션 문서

### 1.1 도메인 작업 영역
아래 `hr`, `inventory`, `admin`은 반드시 최상위 폴더명이라는 뜻이 아니라 작업 책임을 나누기 위한 논리 도메인이다.

| 도메인 | 도메인 키 | 담당 팀원 | 작업 범위 |
| :--- | :--- | :--- | :--- |
| 인사/회계 | `hr` | 서우리 | 인사관리, 급여정산, 매출/지출 등 |
| 구매/재고 | `inventory` | 김슬기 | 재고관리, 발주/입출고, 거래처 관리 등 |
| 관리자/공통 | `admin` | 김민호 | 로그인/보안, 대시보드, 시스템 설정 등 |

실제 디렉터리는 도메인 중심 구조를 따른다.

- `backend/src/main/java/com/dduk/domain/[도메인]/[기능]`
- `frontend/pages/[도메인]`
- `ai-server/services/[도메인]`

새 코드를 추가할 때는 반드시 해당 도메인/기능 하위 패키지에 위치시킨다.

### 1.2 폴더 사용 규칙
1. 본인 도메인이 아닌 코드도 수정할 수는 있지만, 공용 계약이나 다른 담당자 로직에 영향이 있으면 먼저 범위를 공유한다.
2. 여러 도메인에서 같이 쓰는 코드는 `common` 또는 `shared` 성격의 경로에 둔다.
3. 프론트엔드 경로는 구조 변경에 덜 깨지게 일관된 기준으로 관리한다. 절대 경로를 강제하기보다 프로젝트 내 라우팅 규칙 하나로 통일한다.
4. 문서와 소스 파일은 기본적으로 UTF-8 인코딩으로 저장한다.

## 2. Git / PR 규칙

### 2.1 커밋 메시지
커밋 메시지 형식은 강제하지 않는다.  
다만 히스토리를 읽기 쉽게 하려면 아래 형식을 권장한다.

> `type: description`

예시:

- `feat: 직원 근태 관리 API 추가`
- `fix: 재고 차감 시 음수 수량 허용 버그 수정`
- `docs: 협업 가이드라인 검증 규칙 보강`

| 타입 | 의미 |
| :--- | :--- |
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 로직 변경 없는 포맷 수정 |
| `refactor` | 동작 유지 리팩터링 |
| `test` | 테스트 추가 또는 수정 |
| `chore` | 빌드, 설정, 의존성 관리 |

### 2.2 브랜치와 PR
- 기본 브랜치는 직접 푸시하지 않고 기능 브랜치에서 작업한다.
- 브랜치명은 역할이 보이게 짓는다.
- 권장 형식: `feat/...`, `fix/...`, `docs/...`, `chore/...`
- PR에는 변경 목적, 영향 범위, 확인한 검증 결과를 적는다.
- 한 PR에 unrelated 변경을 섞지 않는다.
- 공용 계약 변경 API, DB 스키마, 인증 방식이 있으면 리뷰어가 그 사실을 바로 알 수 있게 PR 본문 첫부분에 적는다.

## 3. 코딩 컨벤션

### 3.1 공통
- 변수명과 함수명은 의미가 드러나는 영문으로 작성한다.
- 들여쓰기는 기본적으로 스페이스 4칸을 사용한다. 단, 이미 파일이나 프레임워크 컨벤션이 다르면 기존 규칙을 따른다.
- 주석은 "무엇을 하는지"보다 "왜 이렇게 했는지"가 코드만으로 드러나지 않을 때만 추가한다.
- 하드코딩된 경로, 비밀값, 환경별 값은 피한다.

### 3.2 Backend (Java/Spring)
- 클래스는 `PascalCase`, 메서드/필드는 `camelCase`를 사용한다.
- 비즈니스 로직은 `Service` 레이어에 두고, Controller는 입출력 처리에 집중한다.
- DTO, Entity, API 응답 모델의 역할을 섞지 않는다.
- 쿼리 성능이 민감한 로직은 fetch 전략과 조회 범위를 코드 리뷰에서 같이 확인한다.

### 3.3 AI/RPA (Python/Flask)
- 함수, 변수, 파일명은 `snake_case`를 사용한다.
- 의존성은 `requirements.txt` 또는 프로젝트에서 정한 파일로 관리한다.
- 모델 키, 외부 API 키, 로그인 정보는 `.env` 또는 별도 설정 계층으로 관리한다.
- 자동화 스크립트는 대상 사이트 변경에 취약하므로 셀렉터와 실패 로그를 같이 관리한다.

### 3.4 Database (MySQL)
- 테이블명과 컬럼명은 `snake_case`를 사용한다.
- 기본 PK 컬럼명은 `id`를 우선한다.
- FK, unique, index는 이름 규칙을 일관되게 맞춘다.
- 문자열 상태값이 늘어날 수 있으면 enum 고정보다 확장 가능성을 먼저 검토한다.

## 4. API 설계
- 모든 API 경로는 기본적으로 `/api/v1/`로 시작한다.
- 리소스명은 복수형을 우선한다.
- HTTP 메서드는 의미에 맞게 쓴다. `GET` 조회, `POST` 생성, `PUT/PATCH` 수정, `DELETE` 삭제.

### 4.1 성공 응답 예시
```json
{
  "status": "success",
  "data": {},
  "message": "요청이 완료되었습니다."
}
```

### 4.2 에러 응답 예시
```json
{
  "status": "error",
  "message": "잘못된 요청입니다.",
  "code": "INVALID_REQUEST"
}
```

### 4.3 API 세부 규칙
- HTTP 상태코드는 실제 결과와 맞게 반환한다. 성공인데 200만 고정하지 않는다.
- 검증 실패는 400 계열, 인증 실패는 401, 권한 없음은 403, 미존재는 404, 서버 오류는 500 계열을 사용한다.
- 목록 조회는 pagination 여부를 명시하고, `page`, `size`, `sort` 같은 파라미터 이름을 일관되게 유지한다.
- 프론트가 사용자에게 그대로 노출할 수 없는 내부 예외 메시지나 스택 트레이스는 응답에 담지 않는다.

## 5. 보안 규칙
- 비밀값, 토큰, 비밀번호, 개인키를 코드나 문서에 하드코딩하지 않는다.
- 외부 입력은 경계에서 검증하고, DB 접근은 파라미터 바인딩이나 안전한 ORM 패턴을 사용한다.
- 인증이 필요한 기능은 인증 여부와 권한 여부를 둘 다 확인한다.
- 파일 업로드, HTML 렌더링, 외부 URL 호출은 악성 입력 가능성을 먼저 본다.
- 사용자에게 보여주는 에러 메시지에는 내부 경로, SQL, 스택 트레이스, 비밀 설정값을 노출하지 않는다.
- 보안상 위험한 기본값 임시 토큰, wide-open CORS, 테스트용 관리자 계정은 로컬 개발 한정으로만 허용하고 배포 전 제거한다.

## 6. 검증 규칙
- 커밋 전에는 최소한 자신이 수정한 영역의 실행 또는 테스트를 확인한다.
- `backend/` 변경 시: 관련 단위 테스트 또는 애플리케이션 기동 확인
- `frontend/` 변경 시: 화면 렌더링 깨짐 여부와 주요 상호작용 확인
- `ai-server/` 또는 `rpa/` 변경 시: 의존성, 설정, 주요 진입 경로 실행 확인
- 테스트를 못 돌린 경우, 왜 못 돌렸는지와 남은 위험을 PR 또는 작업 기록에 남긴다.
- 문서 수정만 한 경우에도 오탈자, 경로, 명령어 예시가 실제 구조와 맞는지 다시 본다.

## 7. AI와 협업할 때
- AI 작업 기본 기준은 [docs/AI_HARNESS.md](/C:/kmh/dduk/docs/AI_HARNESS.md:1)를 우선 참고한다.
- AI에게 수정을 요청할 때는 대상 파일, 수정 의도, 영향 범위를 최대한 구체적으로 전달한다.
- AI가 만든 결과는 그대로 머지하지 말고 사람이 최종 검토한다.
- AI가 제안한 명령, 설정, 보안 관련 변경은 특히 비밀값 노출과 파괴적 동작 여부를 먼저 확인한다.
- AI가 문서 규칙을 바꾸는 경우에는 기존 프로젝트 구조와 충돌하지 않는지 같이 본다.
